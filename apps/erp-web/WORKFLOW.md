# Регламент разработки и деплоя — MetalPro ERP

## Окружения

| Окружение | Ветка | URL | База данных | Назначение |
|-----------|-------|-----|-------------|------------|
| Production | `main` | http://5.35.92.112 | `erp_metal` | Боевой сервер, работают пользователи |
| Staging | `develop` | http://5.35.92.112:3000 | `erp_metal_staging` | Тестирование перед релизом |

---

## Правила ведения веток

- `main` — только стабильный, проверенный код. **Прямые пуши запрещены.**
- `develop` — интеграционная ветка. Сюда сливаются все фичи перед релизом.
- `feature/название` — новая функциональность (пример: `feature/order-export`)
- `fix/название` — исправление бага (пример: `fix/coating-calculation`)
- `refactor/название` — рефакторинг без изменения поведения

---

## Цикл разработки

### 1. Начало работы
```
git checkout develop
git pull origin develop
git checkout -b feature/название
```

### 2. Разработка и коммиты
- Коммиты маленькие и атомарные
- Формат: `feat: описание`, `fix: описание`, `refactor: описание`, `docs: описание`
- Не коммитить `.env`, токены, пароли, большие бинарные файлы

### 3. Деплой на Staging
```
git push origin feature/название
```
Создать PR: `feature/название` → `develop`

**Перед merge в develop проверить:**
- [ ] Код работает локально
- [ ] Нет конфликтов с develop
- [ ] Миграции написаны корректно (нет DROP, массовых удалений)

После merge в `develop` → GitHub Actions автоматически деплоит на staging (http://5.35.92.112:8080)

### 4. Тестирование на Staging
**Обязательно проверить на http://5.35.92.112:8080:**
- [ ] Основной сценарий фичи работает
- [ ] Не сломаны существующие функции (заказы, материалы, калькуляция)
- [ ] Миграции прошли без ошибок
- [ ] PM2 процесс в статусе `online` (не перезапускается циклично)

### 5. Релиз на Production
Только после успешного тестирования на staging.

Создать PR: `develop` → `main`

**Чеклист перед релизом:**
- [ ] Staging протестирован и стабилен
- [ ] PM2 staging: 0 лишних рестартов за последние 30 минут
- [ ] Описание PR содержит: что изменилось, что протестировано, риски
- [ ] Нет незавершённых фич в develop которые не должны попасть в релиз

После merge в `main` → GitHub Actions автоматически деплоит на production.

## Работа с миграциями

**Разрешено:**
- Создание новых таблиц и колонок
- Добавление индексов
- Изменение nullable полей
- Переименование с сохранением данных

**Запрещено без согласования:**
- `DROP TABLE`, `DROP COLUMN`
- Массовое удаление данных
- Изменение типов колонок с потерей данных
- Миграции без отката (всегда думать: как откатить?)

## Hotfix (срочное исправление прода)

Если критический баг на production и ждать цикл develop нельзя:

```
git checkout main
git pull origin main
git checkout -b fix/критический-баг
# ... исправление ...
git push origin fix/критический-баг
```

Создать PR: `fix/критический-баг` → `main` (деплой на прод)
После — обязательно смержить исправление обратно в `develop`:
PR: `fix/критический-баг` → `develop`

---

## Что контролирует Claude

При каждом изменении кода Claude проверяет:
- Соответствие ветки правилам (не пушим напрямую в main)
- Безопасность миграций
- Что фича протестирована на staging перед релизом
- Статус PM2 после деплоя
- Отсутствие токенов и паролей в коде
